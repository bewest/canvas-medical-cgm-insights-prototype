1. Context and Motivation

Canvas Medical is a modern, API-first EHR designed for primary care and value-based practices. Its distinguishing feature is a Python plugin SDK that allows developers to embed arbitrary clinical logic directly inside the EHR, responding to 650+ workflow events and producing effects that alter what clinicians see and do.

Diabetes, particularly Type 1 and insulin-dependent Type 2, represents a high-complexity, data-rich population where care quality is highly sensitive to whether the right information reaches the clinician at the right moment. Most EHRs treat glucose data as passive lab values. Canvas's event and plugin architecture enables something different: active, contextual decision support surfaced inside the normal clinical workflow.

The cgmencode toolkit (from the rag-nightscout-ecosystem-alignment project) represents several years of production ML work on AID system data from Loop, AAPS, and Trio patients. It includes CGM feature encoding, transformer models, clinical loss functions, event classifiers, pharmacokinetics models, and a settings recommendation pipeline. That signal is directly applicable inside a Canvas plugin.

Core thesis: The combination of Canvas's plugin SDK with deep diabetes ML infrastructure can produce decision-support tools that are (a) embedded where clinicians already work, (b) grounded in real CGM/AID patterns rather than textbook heuristics, and (c) structured to generate the documentation chain required for reimbursement.
Scope decisions:
  • Data source: Nightscout as primary CGM backend (FHIR-capable, open source, deep existing tooling in the rag-nightscout-ecosystem-alignment repo).
  • Patient population: Both T1D on AID systems and insulin-dependent T2D. Primary care sees both; the tool should serve the full spectrum.
  • Primary goal: An open-source Canvas plugin that is also a portfolio artifact demonstrating what can be built at this intersection.
  • ML architecture: Canvas plugin as pure orchestrator. ML inference lives entirely in a separate service that wraps the cgmencode research pipeline and its validated experimental findings (externals/, experiments/). The plugin calls that service over HTTP; it does not embed model weights or training logic.

3.5 Nightscout as the Data Bridge

Nightscout is the natural data source for this prototype. It aggregates CGM readings, device status (IOB, COB, temp basal), and treatments across all major AID systems (Loop, AAPS, Trio, OpenAPS) into a unified JSON API. The rag-nightscout-ecosystem-alignment repo has deep tooling built directly on Nightscout's data schema: ns_fetch.py, ns_patients.py, ns_split.py, and the ns2parquet pipeline.

Integration Architecture

The Canvas plugin does not call Nightscout directly. Instead, a lightweight sidecar service (the "cgmencode inference service") mediates:

  1. A patient-to-Nightscout URL mapping is stored as a plugin secret or in a small service registry.
  2. On demand (chart open, appointment creation, post-visit), the Canvas plugin calls the inference service with a patient identifier.
  3. The inference service fetches Nightscout data, runs the cgmencode pipeline (encoding, metrics, optional ML inference), and returns a structured JSON summary.
  4. The plugin renders that summary as Canvas effects (ProtocolCard, BannerAlert, custom chart section, DocumentReference).
Nightscout for T2D patients: Many insulin-dependent T2D patients do not use AID systems. For these patients, the devicestatus endpoint will be absent or minimal. The pipeline degrades gracefully: glucose-only analysis (TIR, GMI, TBR, variability) is still valuable and sufficient for the care gap and billing use cases. AID-specific features (IOB-based settings recommendations) apply only when devicestatus data is present.

3.6 Where the Code Lives: Deployment Topology

Decisive constraint (verified against canvas-medical/canvas-plugins v0.173.0 source): The Canvas plugin runtime executes plugin code under RestrictedPython with a hard import allowlist (plugin_runner/sandbox.py, ALLOWED_MODULES). A plugin may import only: a curated subset of the standard library (datetime, json, re, dataclasses, decimal, hashlib, base64, uuid, collections.Counter/defaultdict, typing, enum), a short third-party list (requests, pydantic, arrow, rapidfuzz, jwt, django.db.models), and canvas_sdk.*. numpy, scipy, pandas, torch, xgboost, sklearn — and even math and statistics — are NOT importable. Plugins cannot pip-install their own dependencies; installation.py has no per-plugin dependency step. This means none of the cgmencode pipeline can run inside a Canvas plugin. The plugin is strictly an orchestrator.
What this resolves: The sidecar architecture is not just preferable, it is required. The good news, also verified in source: outbound HTTP is fully supported. canvas_sdk.utils.http.Http (allowlisted) wraps requests with GET/POST/PUT/PATCH and parallel batch helpers (batch_get, ThreadPoolExecutor). canvas_sdk.clients.canvas_fhir.CanvasFhir is allowlisted for reading Canvas's own FHIR data. So a plugin can call an external T1Pal service and render the result. The earlier finding still holds and matters: because the production pipeline is pure numpy/scipy/pandas (no torch/GPU), that external service is a trivially cheap, CPU-only Python web app.
On the cgmencode pipeline itself: The production/ decision-support code does not require PyTorch, XGBoost, or a GPU. It is built on numpy, scipy.stats (bootstrap CIs, linear regression), and pandas. The sklearn dependency is a single optional lazy import inside MealMLModel.train() that degrades gracefully when absent. torch and xgboost exist only in the research/training layer (encoder.py, model.py, hypo_safety.py training code), not in the production inference path. So the sidecar is cheap to run, but it must exist: the sandbox forbids even basic numpy in-plugin.

The Three Deployment Options, Re-evaluated

Option A

Canvas Plugin (inline) — viable for metrics, not the full pipeline

A plugin cannot import numpy/scipy/pandas/math or pip-install deps. But the sandbox does expose enough builtins (sum, sorted, min/max, len, abs, round, the ** power operator, comprehensions) to compute real descriptive CGM metrics in pure Python: TIR/TBR/TAR, mean, GMI, CV (std via x**0.5), and time-of-day percentiles. Plus datetime math and Django ORM aggregates. What it cannot do: the numpy/scipy physics simulation, bootstrap CIs, regression, and ML — i.e. settings extraction and parameter recommendation.

Option B (primary)

T1Pal Sidecar Service

A FastAPI service co-located with T1Pal infrastructure. Data is already there; no PHI leaves T1Pal's HIPAA boundary. Runs the full pipeline on demand or nightly batch. Canvas plugin calls it via canvas_sdk.utils.http.Http and renders cached results. Best for: full pipeline, waterfall, deconfounding, settings optimizer.

Option C (recommended)

Hybrid

Fast path: care-gap rules and threshold alerts run inside Canvas using only allowlisted modules (datetime + ORM aggregates), no external call. Deep path: settings recommendations and pattern analysis call the T1Pal service asynchronously; results cached as a Canvas DocumentReference before the visit. Both latency and depth.

What Needs torch/XGBoost (Research Layer Only)

The following are research/training tools, not production inference. They run on the sidecar (or offline), never inside Canvas:

  • encoder.py: tensor conversion for training datasets (torch)
  • model.py, toolbox.py: Transformer AE/VAE architecture definitions (torch)
  • hypo_safety.py: training loss functions and fine-tuning (torch)
  • event_classifier.py: XGBoost training pipeline (xgboost)
  • override_recommender.py: counterfactual forecasting via trained checkpoints (torch)

The production hypo predictor uses hardcoded regression coefficients derived from those experiments; it does not load a model file at runtime. But it still uses numpy, so it runs on the sidecar, not in the plugin.

Feasibility re-assessment: Two independent facts, both verified in source. (1) The cgmencode production pipeline is a pure statistics package: deployable as a standard CPU-only Python web service on a $10/month VPS, no GPU, no model weights, no torch. (2) The Canvas plugin sandbox forbids all scientific Python, so that service is mandatory, not optional. The plugin is a thin orchestration + presentation layer (allowlisted modules + SDK effects). Net: lower ML-ops complexity than expected, but a clean two-process split is forced by the sandbox rather than chosen.

Should We Mirror CGM Data into Canvas as FHIR?

Writing data into Canvas is well supported, and ingestion is a Canvas-blessed plugin pattern: the SDK ships example_patient_sync and canvas_fhir_client examples. There are two write paths:

PathMechanismBest for
Observation effect canvas_sdk.effects.observation.Observation(...) returned from compute(); creates/updates an Observation record in-process (value, units, codings, components, effective_datetime). Writing a small number of derived records (e.g. a weekly TIR/GMI summary) as a side effect of an event the plugin already handles.
CanvasFhir client canvas_sdk.clients.canvas_fhir.CanvasFhir(client_id, client_secret) with .create() / .read() / .search() against fumage-<customer>.canvasmedical.com. Programmatic FHIR writes, including from an external integration service / sidecar.
SimpleAPI inbound route SimpleAPI + APIKeyAuthMixin exposing @api.post(...); an external system pushes data in and the handler writes records via effects. Event-driven push from Nightscout / a device cloud (one reading or one summary per call). This is the canonical "external system is point of origination" shape.
Practical recommendation: do NOT mirror the raw CGM stream into Canvas.
  • Volume: CGM is ~288 readings/day. 14 days × a 1,000-patient cohort is ~4M Observations per fortnight. Canvas is a clinical and billing record, not a time-series database. A raw dump bloats the chart and no clinician wants 288 glucose values per day in the record.
  • Plugins time out: a multi-thousand-row backfill loop inside a chart-open handler is the wrong tool. Bulk or continuous ingestion belongs in the sidecar via the FHIR API, or as event-driven single-reading pushes via a SimpleAPI webhook.
  • Write derived artifacts, not the stream: a periodic CGM summary Observation (TIR/GMI/CV) and a DocumentReference (AGP report). These are clinically meaningful and billable (CPT 95251). The raw 5-minute series stays in Nightscout / the sidecar, its natural home.

Decision guide: when to write into Canvas

GoalWrite to Canvas?What to write
Render CGM display / triage at point of care No (POC) Fetch-on-demand from NS, compute in-plugin, render. No persistence needed.
Billing documentation (95251 / RPM) Yes One DocumentReference (AGP report) + a summary Observation per period.
Native Canvas protocols / quality measures should see the data Yes Summary Observations (e.g. latest GMI, TIR) — codified, low cardinality.
Raw 5-minute stream queryable inside Canvas Almost never Keep in Nightscout / sidecar; expose via the inference API instead.

For the POC: no mirroring. The plugin fetches on demand and renders. A nice stretch goal that demonstrates the write path without abusing the record: emit a single weekly CGM summary Observation plus a DocumentReference, gated behind a note-signed or appointment event.

4. Opportunity Map

Below are specific, concrete things that become possible by combining diabetes ML expertise with the Canvas API surface.

A. Automated CGM Summary at Point of Care

When a clinician opens the chart of a patient with diabetes (event: PATIENT_CHART_SUMMARY__SECTION_CONFIGURATION or chart open), the plugin fetches the patient's recent CGM data from a connected source (Nightscout, Tidepool, Dexcom API), runs the FixtureEncoder pipeline and metrics.py to compute TIR, GMI, time below range, and glucose variability, then renders a custom summary section: no manual data entry, no portal switching.

What this eliminates: The 5-10 minutes clinicians currently spend pulling up a separate app or asking the patient to show their phone. The data is present before the visit begins.

B. Hypoglycemia Safety Protocol Card

Using hypo_safety.py output and the ADD_OR_UPDATE_PROTOCOL_CARD effect, surface a card whenever a patient's recent CGM data shows: TBR (time below range) above 4% of day, more than 1 Level 2 hypo (below 54 mg/dL) in the past 14 days, or a hypoglycemia unawareness pattern. The card offers one-click actions to order a glucagon prescription, refer to endocrinology, or adjust the safety targets note.

C. Pre-Visit AID Settings Review

On appointment creation (event: APPOINTMENT__PRE_CREATE or created), trigger an async job that runs the override_recommender pipeline on the patient's 30-day CGM / AID device status history. Attach a structured settings summary as a DocumentReference and surface a protocol card: "Suggested insulin settings adjustments available for review." The clinician sees evidence-based recommendations before the patient walks in, rather than spending the visit manually reading pump logs.

D. Care Gap Automation (Diabetes Quality Measures)

The following care gaps are high-value and straightforward to automate with Canvas protocol cards, triggered by condition presence (ICD E10/E11) and lab result age:

  • HbA1c overdue (HEDIS: within 6 months for controlled, 3 months for uncontrolled)
  • Annual kidney function panel (eGFR + UACR): nephropathy screening
  • Annual retinal exam referral
  • Annual foot exam (monofilament) documentation
  • Blood pressure at goal (<130/80 for most diabetes patients)
  • Statin therapy for patients over 40 with diabetes
  • CGM or SMBG frequency adequate for insulin users

Each card offers a one-click action to order the lab or create the referral, directly generating the documentation needed for quality measure credit and physician reimbursement.

E. Post-Visit Automated Documentation

After an encounter closes (event: NOTE_STATE_CHANGE_EVENT_UPDATED, status to signed), generate a structured CGM interpretation note appended to the chart as a DocumentReference. This creates the documentation chain that enables billing for CPT 95251 (CGM professional interpretation) and supports RPM billing for the 99454 device data transmission codes.

F. Predictive Hypoglycemia Banner

Using the event classifier trained on pre-event windows, when a patient's CGM data (ingested continuously via SimpleAPI webhook) shows a pattern consistent with impending hypoglycemia, trigger an async ADD_BANNER_ALERT effect so the clinician (or care coordinator reviewing the panel) sees a real-time risk flag. This is the most technically ambitious of the near-term opportunities and requires a continuous data feed and a deployed inference service.

G. Population Health Panel for AID Patients

Using the Canvas FHIR API from outside the instance, build a population-level cohort view of all patients on AID therapy: TIR distribution, HbA1c trend, care gap completion rate. This can be built as an external application using the FHIR API, feeding Canvas Tasks back for outreach prioritization.

5. Proof-of-Concept Scope: A Self-Contained Plugin

The feasibility line, drawn precisely: The full cgmencode production tooling (settings extraction, parameter recommendation) cannot live in a plugin, because it depends on numpy/scipy physics simulation, bootstrap confidence intervals, and regression. But a descriptive analysis + display + triage plugin can be fully self-contained in pure Python, with no sidecar and no external ML. That self-contained slice is exactly the right shape for an open-source portfolio piece.

What an Open-Source Portfolio Piece Can Honestly Be

An OSS portfolio plugin is necessarily narrower than the commercial-grade settings engine, and that is fine: it should demonstrate platform fluency and clinical judgment, not ship an unvalidated dosing recommender. The POC proves you can take real CGM data, compute the right clinical metrics, and surface them where a clinician works, end to end, inside Canvas. The heavy statistical pipeline is referenced as the "Phase 2 sidecar," not built into the demo.

Metrics in Pure Python (no numpy, runs in-plugin)

def cgm_metrics(sgv: list[float]) -> dict:
    """All standard CGM metrics from a list of mg/dL readings.
    Uses only sandbox-allowed builtins: sum, len, sorted, ** operator."""
    n = len(sgv)
    if n == 0:
        return {}
    mean = sum(sgv) / n
    var = sum((x - mean) ** 2 for x in sgv) / n
    std = var ** 0.5                              # sqrt without the math module
    pct = lambda p: sorted(sgv)[min(n - 1, int(p * n))]

    in_range = sum(1 for x in sgv if 70 <= x <= 180)
    below    = sum(1 for x in sgv if x < 70)
    very_low = sum(1 for x in sgv if x < 54)
    above    = sum(1 for x in sgv if x > 180)

    return {
        "mean": round(mean, 1),
        "gmi": round(3.31 + 0.02392 * mean, 1),  # GMI %, Bergenstal 2018
        "cv": round(std / mean * 100, 1),        # %CV; >=36% = high variability
        "tir": round(100 * in_range / n, 1),
        "tbr": round(100 * below / n, 1),
        "tbr_l2": round(100 * very_low / n, 1),
        "tar": round(100 * above / n, 1),
        "p25": pct(0.25), "p50": pct(0.50), "p75": pct(0.75),
    }

This function uses nothing the RestrictedPython sandbox forbids. It can run directly inside the plugin's compute() on glucose pulled from Nightscout (Http.get) or from Canvas FHIR Observations. This is the entire numerical core of the POC.

Explicitly Out of POC Scope (Phase 2, sidecar + validation)

  • Settings extraction: ISF/CR/basal optimization (settings_optimizer.py, advisor/) — numpy/scipy physics simulation and bootstrap CIs. Sidecar only.
  • Natural-experiment detection and counterfactual TIR simulation.
  • ML extras: sklearn meal-timing prediction; torch pattern retrieval.
  • Any active dosing/parameter recommendation surfaced to a clinician as actionable — this carries clinical-validation and liability requirements beyond a portfolio artifact, and should be gated behind the validated sidecar.
Why this is the right POC: It is genuinely self-contained (one plugin, pure Python, installable in a Canvas sandbox with no extra infrastructure), it exercises the full SDK surface that matters (events, custom chart sections, protocol cards, banner alerts, SimpleAPI, Data module, Http client), and it tells an honest story: descriptive analytics and triage today; the validated statistical engine as a clearly-scoped Phase 2.

from canvas_sdk.effects import Effect
from canvas_sdk.effects.protocol_card import ProtocolCard, Recommendation
from canvas_sdk.effects.banner_alert import AddBannerAlert
from canvas_sdk.events import EventType
from canvas_sdk.handlers import BaseHandler
from canvas_sdk.utils.http import Http
from canvas_sdk.v1.data import Condition


class DiabetesCGMHandler(BaseHandler):
    """Surface CGM summary and care gaps on chart open."""

    RESPONDS_TO = EventType.Name(EventType.PATIENT_CHART_SUMMARY__SECTION_CONFIGURATION)

    # ICD-10 prefixes for diabetes (E10.* T1D, E11.* T2D)
    DM_PREFIXES = ("E10", "E11")

    def compute(self) -> list[Effect]:
        patient_id = self.context["patient"]["id"]

        # Diabetes diagnosis check via Data module (Django ORM, allowlisted)
        has_dm = (
            Condition.objects.active()
            .for_patient(patient_id)
            .filter(codings__code__startswith="E1")
            .exists()
        )
        if not has_dm:
            return []

        # All cgmencode compute happens in the external T1Pal sidecar.
        # The plugin only orchestrates via the allowlisted Http client.
        http = Http()
        resp = http.get(
            f"{self.secrets['CGM_SERVICE_URL']}/summary/{patient_id}",
            headers={"Authorization": f"Bearer {self.secrets['CGM_SERVICE_KEY']}"},
        )
        if resp.status_code != 200:
            return []

        summary = resp.json()              # {tir, tbr, gmi, settings_recs: [...]}
        tir = summary.get("tir_percent", 0)
        tbr = summary.get("tbr_percent", 0)
        gmi = summary.get("gmi")

        effects: list[Effect] = []

        # Hypo safety banner if Time Below Range is critical.
        # NOTE: AddBannerAlert.narrative is capped at 90 characters.
        if tbr > 4.0:
            effects.append(
                AddBannerAlert(
                    patient_id=patient_id,
                    key="cgm-tbr-alert",
                    narrative=f"Time Below Range {tbr:.1f}% (goal <4%). Review hypo safety plan.",
                    placement=[AddBannerAlert.Placement.CHART],
                    intent=AddBannerAlert.Intent.WARNING,
                ).apply()
            )

        # Protocol card for TIR below target, with actionable recommendations.
        if tir < 70:
            card = ProtocolCard(
                patient_id=patient_id,
                key="cgm-tir-below-target",
                title="CGM Time-in-Range Below Target",
                narrative=(
                    f"Current TIR {tir:.0f}% (goal >70%). GMI {gmi or 'N/A'}. "
                    "Consider settings review or therapy intensification."
                ),
                status=ProtocolCard.Status.DUE,
            )
            card.recommendations.append(
                Recommendation(title="Order HbA1c", button="Order")
            )
            card.recommendations.append(
                Recommendation(title="Refer to Endocrinology", button="Refer")
            )
            effects.append(card.apply())

        return effects
Verified API notes:
  • ProtocolCard requires patient_id (or patient_filter) and a unique key; recommendations are Recommendation objects (or built from a command via command.recommend(title=, button=)), not raw dicts.
  • AddBannerAlert.narrative is hard-capped at 90 characters by a Pydantic validator; placement is a list of Placement enums and intent is an Intent enum (info/warning/alert).
  • Outbound HTTP must use canvas_sdk.utils.http.Http (wraps requests); httpx is not allowlisted. Parallel fan-out is available via batch_get + ThreadPoolExecutor.
  • Patient/condition context comes from self.context and the Data module (Condition.objects.active().for_patient(id)), which also supports ValueSet lookups for more robust diabetes cohorting than raw ICD prefixes.

8. Reimbursement and Documentation Chain

Healthcare digital tools succeed when they create a chain of documentation that results in reimbursement for physicians while reducing, not adding to, their workload. The following billing codes are directly supported by the architecture above.

CodeDescriptionData gateWhat This System Generates
CPT 95251 Ambulatory CGM data interpretation and report, professional ≥72 h of CGM data DocumentReference (AGP + interpretation) + summary Observations (TIR, GMI, mean) + clinician review/sign prompt
CPT 99453 RPM: patient setup and education for remote monitoring device One-time, at enrollment Documented via Task + Note at device enrollment
CPT 99454 RPM: device supply with daily recordings; 30-day period ≥16 days with readings / 30 days In-plugin count of distinct days with data; monthly summary + readiness flag when the 16-day threshold is met
CPT 99457 RPM: 20+ min of interactive communication per 30-day period ≥20 min documented/month Canvas Task + time-tracking for care-coordinator outreach (workflow feature; stubbed/flagged in POC)
HEDIS / STAR quality measures Diabetes HbA1c control, eye exam, kidney health Per-measure logic Protocol card completions generate codified orders / referrals that count toward measure numerators

Billing-Readiness in the POC

Per the data-lifecycle decision (don't mirror raw streams, do capture statistics and documentation), the reimbursement chain is explicitly in scope for the proof of concept. It is low-volume, high-value, and computable in pure Python: exactly the kind of differentiated capability worth showing. The POC produces three things, each gated by the data-sufficiency checks above so an artifact is never emitted on insufficient data:

  1. A signed-ready report (DocumentReference): the AGP image/summary plus a short structured interpretation. This is the artifact the billing provider reviews and signs for CPT 95251. Typed with an appropriate report LOINC; raw stream not included.
  2. Codified summary Observations: TIR, GMI, mean glucose, %CV as low-cardinality Observations (appropriate LOINC, verified against the practice's coding setup) so the chart, native protocols, and quality measures can consume them.
  3. A review-and-sign prompt (ProtocolCard / Task): surfaces the suggested CPT code and the data-sufficiency status, and routes the clinician to sign. The human sign-off is the step that converts the documentation into a billable service, exactly the step that should require a human.
Honest boundary: The POC generates the documentation artifact and the billing-readiness signals. It does not submit claims, and it is not a billing engine. Actual reimbursement still requires clinician review and signature, an order / established patient relationship, payer coverage, and the practice's billing configuration. The plugin's job is to make the documentation effortless and correctly gated, not to adjudicate. Note also that CGM (95249/95250/95251) and RPM (99453/99454/99457) codes carry bundling rules; the POC should flag the eligible code(s) and let the practice's billing logic decide, rather than co-bill naively.
Key friction reduction: The goal is that the physician sees a protocol card with one-click actions. Clicking "Order HbA1c" generates both the order and the supporting documentation. The chain from CGM data to billable service is fully automated, except for the final clinician sign-off, which is exactly what should require a human.

9. Open Questions and Refinement Areas

Resolved Scope Decisions

  1. Data source: Nightscout (primary). Tidepool and Dexcom API are stretch targets once the Nightscout path is validated.
  2. Patient population: Both T1D on AID and insulin-dependent T2D. AID-specific features (IOB/COB/settings review) activate only when devicestatus data is present; glucose-only features apply universally.
  3. Primary goal: Open-source Canvas plugin that doubles as a portfolio artifact.
  4. ML architecture: Plugin is a pure orchestrator. cgmencode inference service is a separately hosted Python service respecting the validated experimental findings in externals/ and experiments/.

Open Questions: Next Round

  1. Nightscout auth model: Should the prototype assume each patient supplies their own Nightscout URL + API secret, or is there a shared deployment model (e.g., T1Pal hosted instance) that simplifies onboarding?
  2. cgmencode checkpoint availability: Which validated model checkpoints from the experimental pipeline are stable enough to serve via the inference API? The override_recommender and metrics pipeline are rule-based; which ML models (event classifier, transformer AE) have sufficient validation to expose?
  3. Canvas sandbox access: Do we have (or should we request) a Canvas developer sandbox to test the plugin interactively?
  4. Open-source licensing: The cgmencode repo and its externals need a license audit before any code is surfaced in a public plugin repo. What license is appropriate?
  5. Portfolio framing: Should the plugin README and docs explicitly call out the connection to the Canvas Product Experience Builder role, or stay neutral so it reads as a standalone open-source project?

Resolved by Source Review (canvas-medical/canvas-plugins v0.173.0)

  • Plugin dependency model: RESOLVED. Plugins run under RestrictedPython with a hard import allowlist (plugin_runner/sandbox.py). No numpy/scipy/ pandas/math; no per-plugin pip installs. All cgmencode compute must be external. This is now reflected throughout the topology section.
  • Outbound HTTP from a plugin: RESOLVED. canvas_sdk.utils.http.Http (allowlisted, wraps requests) supports GET/POST/PUT/PATCH plus parallel batch helpers. A plugin can call the T1Pal sidecar directly. No Canvas partnership is required for a plugin to call an externally hosted service the developer controls.
  • SimpleAPI webhook endpoints: RESOLVED. canvas_sdk.handlers.simple_api provides SimpleAPI / SimpleAPIRoute with @get/@post/@put/@delete route decorators, a Request object (incl. multipart/file parts), and handlers returning list[Response | Effect]. This is the mechanism for receiving a Nightscout push or a sidecar callback inside Canvas.
  • Effect APIs: RESOLVED. ProtocolCard/AddBannerAlert signatures verified; the plugin skeleton above now matches the real classes (including the 90-char banner narrative cap and required key fields).
  • Data access: RESOLVED for reads. The Data module exposes Condition, Patient, labs, medications via Django ORM with .for_patient() and ValueSet lookups. Read access is to the Canvas record, not limited to plugin-created rows.

Still Open

  • Nightscout auth model: Per-patient URL + token, or a shared T1Pal-hosted registry? Affects onboarding UX and where the patient↔Nightscout mapping is stored (plugin secret vs. sidecar registry).
  • FHIR Observation bulk ingest: If we choose to mirror CGM history into Canvas as Observations (vs. keeping it only in the sidecar), confirm batch/bundle support to avoid per-reading POSTs. The current design keeps CGM data in the sidecar and only writes summaries as DocumentReference, sidestepping this.
  • cgmencode checkpoint availability: The rule-based + statistical production pipeline (metrics, hypo_predictor, settings_advisor/optimizer) needs no checkpoints. Only the optional ML extras (sklearn meal model; torch pattern retrieval) do. Decide whether to expose those in v1.
  • Canvas developer sandbox: Request an instance to test plugin install, effects rendering, and the SimpleAPI webhook end-to-end.
  • Open-source licensing: License audit of cgmencode and its externals/ before any code is surfaced in a public plugin repo.
  • Portfolio framing: Should the plugin README explicitly reference the Canvas role, or read as a neutral standalone open-source project?
